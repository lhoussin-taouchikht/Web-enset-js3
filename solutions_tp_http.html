TP 2 : Maîtrise de cURL - Solutions

2.1 Question : Différence entre -i et -v

Option -i (include)

  • Affiche les headers de réponse HTTP
  • Montre uniquement la réponse du serveur
  • Plus simple et concis
  • Utile pour voir rapidement les codes de statut et headers

Option -v (verbose)

  • Mode debug complet
  • Affiche les headers de requête ET de réponse
  • Montre les détails de connexion SSL/TLS
  • Affiche les négociations de protocole
  • Utile pour le débogage approfondi
En résumé : Utilisez -i pour voir simplement la réponse, et -v pour un débogage complet avec tous les détails de la communication.

Exercice avancé - Solution

Commande cURL complète qui remplit toutes les conditions :

curl -i -X POST \
  -H "Content-Type: application/json" \
  -H "X-Custom-Header: MonHeader" \
  -d '{"action": "test", "value": 42}' \
  https://httpbin.org/post

Explication des options :

  • -i : Affiche les headers de réponse
  • -X POST : Spécifie la méthode HTTP POST
  • -H : Ajoute un header personnalisé (peut être utilisé plusieurs fois)
  • -d : Définit le corps (body) de la requête

TP 3 : API REST avec JavaScript - Solutions

Exercice pratique : fetchWithRetry

Fonction qui effectue des tentatives multiples en cas d'erreur 5xx :

async function fetchWithRetry(url, options = {}, maxRetries = 3) {
  let lastError;
  
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const response = await fetch(url, options);
      
      // Si erreur 5xx et il reste des tentatives, on réessaie
      if (response.status >= 500 && response.status < 600) {
        if (attempt < maxRetries) {
          console.log(`Erreur ${response.status}, tentative ${attempt + 1}/${maxRetries}`);
          // Attendre 1 seconde avant de réessayer
          await new Promise(resolve => setTimeout(resolve, 1000));
          continue;
        }
        throw new Error(`Erreur serveur ${response.status} après ${maxRetries} tentatives`);
      }
      
      // Pour les autres erreurs (4xx), on ne réessaie pas
      if (!response.ok) {
        throw new Error(`HTTP ${response.status}: ${response.statusText}`);
      }
      
      // Succès : on retourne la réponse
      return response;
      
    } catch (error) {
      lastError = error;
      
      // Erreurs réseau : on réessaie
      if (attempt < maxRetries) {
        console.log(`Erreur réseau, tentative ${attempt + 1}/${maxRetries}`);
        await new Promise(resolve => setTimeout(resolve, 1000));
        continue;
      }
    }
  }
  
  // Si on arrive ici, toutes les tentatives ont échoué
  throw lastError;
}

Exemple d'utilisation :

// Utilisation simple
fetchWithRetry('https://api.example.com/data')
  .then(response => response.json())
  .then(data => console.log(data))
  .catch(error => console.error('Échec après tentatives:', error));

// Avec options personnalisées et 5 tentatives
fetchWithRetry('https://api.example.com/post', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ data: 'test' })
}, 5)
  .then(response => response.json())
  .then(console.log);
Points clés de la solution :
  • Réessaie uniquement pour les erreurs 5xx (serveur) et les erreurs réseau
  • Ne réessaie PAS pour les erreurs 4xx (client)
  • Attend exactement 1 seconde entre chaque tentative
  • Lève une erreur appropriée après épuisement des tentatives

Détails des headers GitHub

Strict-Transport-Security (HSTS)

max-age=31536000; includeSubDomains; preload

Force HTTPS pendant 1 an, inclut tous les sous-domaines, et est dans la liste de préchargement des navigateurs.

X-Frame-Options

deny

Empêche complètement l'affichage du site dans une iframe (protection contre le clickjacking).

X-Content-Type-Options

nosniff

Empêche le navigateur de deviner le type MIME (protection contre les attaques MIME).

Content-Security-Policy

default-src 'none'; base-uri 'self'; ...

Définit strictement les sources autorisées pour charger les ressources.

Referrer-Policy

origin-when-cross-origin, strict-origin-when-cross-origin

Contrôle les informations envoyées dans le header Referer.

Configuration serveur (Node.js/Express)

const express = require('express');
const app = express();

// Cache pour les fichiers CSS (longue durée - 1 an)
app.use('/style.css', (req, res, next) => {
  res.setHeader('Cache-Control', 'public, max-age=31536000, immutable');
  res.setHeader('ETag', 'w/"css-v1"');
  next();
});

// Cache pour les images (longue durée - 1 an)
app.use('/image.jpg', (req, res, next) => {
  res.setHeader('Cache-Control', 'public, max-age=31536000, immutable');
  res.setHeader('ETag', 'w/"img-v1"');
  next();
});

// Cache pour JavaScript (durée moyenne - 1 jour)
app.use('/script.js', (req, res, next) => {
  res.setHeader('Cache-Control', 'public, max-age=86400');
  res.setHeader('ETag', 'w/"js-v1"');
  next();
});

// HTML sans cache (toujours frais)
app.use('/', (req, res, next) => {
  res.setHeader('Cache-Control', 'no-cache');
  next();
});

app.use(express.static('public'));
app.listen(3000, () => console.log('Serveur démarré sur le port 3000'));

Recommandations de cache par type de fichier

Type de fichier Cache-Control Raison
HTML no-cache ou max-age=0 Contenu dynamique, doit être validé à chaque fois
CSS/JS versionné public, max-age=31536000, immutable Fichiers avec hash dans le nom (style.a3f2b1.css), jamais modifiés
Images public, max-age=604800 Changent rarement (cache 1 semaine)
API JSON no-store Données sensibles ou en temps réel, jamais mises en cache
Polices (fonts) public, max-age=31536000 Ne changent jamais une fois publiées
Bonnes pratiques :
  • Utilisez le versioning dans les noms de fichiers (style.v2.css)
  • Ajoutez l'attribut immutable pour les fichiers versionnés
  • Combinez max-age avec ETag pour validation conditionnelle
  • Utilisez public pour autoriser le cache des CDN

Exercice 2 : Questions théoriques - Réponses détaillées

Question 1 : Différence entre no-cache et no-store

no-cache

  • Le navigateur peut stocker la ressource en cache
  • MAIS doit valider avec le serveur avant chaque utilisation
  • Utilise les headers ETag ou Last-Modified pour la validation
  • Si la ressource n'a pas changé → serveur répond 304 Not Modified
  • Économie de bande passante (pas de retransmission du contenu)

no-store

  • Aucune mise en cache autorisée
  • Ni dans le cache du navigateur, ni dans les caches intermédiaires (proxies, CDN)
  • Chaque requête télécharge la ressource complète
  • Utilisé pour les données sensibles (bancaires, médicales, etc.)
  • Plus sécurisé mais plus coûteux en bande passante
Exemple pratique :
Cache-Control: no-cache → Page d'actualités (vérifier si nouvelle version existe)
Cache-Control: no-store → Relevé bancaire (ne jamais mettre en cache)

Question 2 : Pourquoi POST n'est-il pas idempotent ?

Définition d'idempotence : Une opération est idempotente si l'exécuter plusieurs fois produit le même résultat qu'une seule exécution.

POST n'est pas idempotent car :

  • Chaque requête POST peut créer une nouvelle ressource
  • L'état du serveur change différemment à chaque requête identique
  • Exemple : POST /commandes envoyé 2 fois → 2 commandes distinctes créées
  • Aucune garantie que le résultat soit identique

Méthodes idempotentes (pour comparaison) :

  • GET : Lire 10 fois → même résultat
  • PUT : Mettre à jour 10 fois avec les mêmes données → même état final
  • DELETE : Supprimer 10 fois → même état final (déjà supprimé)
Attention : C'est pourquoi les navigateurs demandent confirmation pour renvoyer un formulaire POST, mais pas pour rafraîchir une page GET.

Question 3 : Que se passe-t-il si le serveur renvoie un code 301 ?

Code 301 - Moved Permanently (Déplacé définitivement)

Comportement :

  • Indique que la ressource a été déplacée définitivement vers une nouvelle URL
  • Le serveur renvoie la nouvelle URL dans le header Location
  • Le navigateur redirige automatiquement vers la nouvelle URL
  • Les moteurs de recherche mettent à jour leurs index avec la nouvelle URL
  • Le navigateur peut mettre en cache cette redirection

Exemple :

HTTP/1.1 301 Moved Permanently
Location: https://www.example.com/nouvelle-page
Content-Length: 0

Différence avec 302 (Moved Temporarily) :

Code Durée Cache SEO
301 Permanent Oui Transfère le référencement
302 Temporaire Non Garde l'ancienne URL

Question 4 : À quoi sert le header Origin ?

Le header Origin :

  • Indique l'origine de la requête (protocole + domaine + port)
  • Utilisé pour le mécanisme CORS (Cross-Origin Resource Sharing)
  • Permet au serveur de décider d'autoriser ou non la requête
  • Automatiquement ajouté par le navigateur pour les requêtes cross-origin
  • Ne peut pas être modifié par JavaScript (sécurité)

Exemple :

Origin: https://example.com

Fonctionnement CORS :

  1. Site A (https://site-a.com) fait une requête vers API B (https://api-b.com)
  2. Navigateur ajoute : Origin: https://site-a.com
  3. API B vérifie l'origine et répond avec : 
    Access-Control-Allow-Origin: https://site-a.com
  4. Navigateur autorise la réponse si les origines correspondent
Sécurité : Sans CORS, un site malveillant pourrait lire vos données sur d'autres sites où vous êtes connecté.

Question 5 : Pourquoi utiliser HttpOnly sur les cookies de session ?

L'attribut HttpOnly :

  • Empêche l'accès au cookie via JavaScript (document.cookie)
  • Le cookie est uniquement envoyé dans les requêtes HTTP
  • Protège contre les attaques XSS (Cross-Site Scripting)
  • Essentiel pour les cookies de session et d'authentification

Exemple de cookie sécurisé :

Set-Cookie: sessionId=abc123xyz; HttpOnly; Secure; SameSite=Strict

Attributs de sécurité combinés :

Attribut Protection Description
HttpOnly XSS Pas d'accès JavaScript au cookie
Secure Interception Cookie envoyé uniquement en HTTPS
SameSite CSRF Cookie non envoyé dans requêtes cross-site

Scénario d'attaque sans HttpOnly :

  1. Attaquant injecte du JavaScript malveillant (XSS)
  2. Le script lit le cookie de session : document.cookie
  3. Le cookie est envoyé au serveur de l'attaquant
  4. L'attaquant peut usurper l'identité de la victime
Avec HttpOnly : Le JavaScript ne peut pas lire le cookie → attaque XSS neutralisée pour le vol de session.
